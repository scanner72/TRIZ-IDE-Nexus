# ТЕХНИЧЕСКОЕ ЗАДАНИЕ (PRD/TECH SPEC)
## Проект: TRIZ-IDE Nexus (Кастомная инженерная среда разработки на базе форка Void)

---

## 1. ОБЩИЕ ПОЛОЖЕНИЯ И НАЗНАЧЕНИЕ

**TRIZ-IDE Nexus** — это специализированная суверенная среда разработки (IDE), создаваемая путем глубокой модификации и ребрендинга открытого форка VS Code — **Void**. 

В отличие от стандартных AI-ассистентов (Cursor, Copilot), нацеленных на потоковое автозаполнение кода, Nexus принудительно внедряет инженерные алгоритмы **ТРИЗ (Теория решения изобретательских задач)** и **АРИЗ-85-В** на этапе проектирования архитектуры. Цель системы — заставить ИИ-агент и разработчика устранять архитектурные противоречия *до* генерации исходного кода, обеспечивая максимальный КПД и лаконичность программных решений.

---

## 2. АРХИТЕКТУРА И СТЕК ТЕХНОЛОГИЙ

Система реализует принцип разделения функций (Прием ТРИЗ №2: Вынесение) и состоит из трех независимых слоев:

1. **Клиентский слой (UI/Editor):** Локальная сборка форка Void. Полный ребрендинг метаданных, изоляция конфигурационных путей окружения и кастомизация интерфейса перехвата ИИ-запросов.
2. **Логический слой оркестрации (TRIZ MCP Server):** Локальный сервис на Python/TypeScript, выступающий в роли «ТРИЗ-инженера». Реализует пошаговые проверки по АРИЗ.
3. **Слой долгосрочной памяти (Context Layer):** Интеграция с базой знаний через **MemoryBridge**. Содержит формализованные правила, 40 изобретательских приемов Альтшуллера и архивы фонда Г.С. Альтшуллера.

## 3. ЭТАПЫ РАЗРАБОТКИ И РЕБРЕНДИНГА (ROADMAP)

### Этап 1: Изоляция и Ребрендинг Форка Void
**Цель:** Полное переименование среды разработки и исключение конфликтов с другими IDE в системе.

*   **Модификация `package.json`:**
    *   Изменение `"name"` на `"triz-ide-nexus"`.
    *   Изменение `"displayName"` на `"TRIZ-IDE Nexus"`.
*   **Модификация `product.json`:**
    *   Переименование полей `"nameShort"` и `"nameLong"` на `"Nexus"`.
    *   Установка `"applicationName"` в `"nexus"`.
    *   Изменение `"dataFolderName"` на `".nexus-ide"`. Это критически важно для изоляции глобального кэша, расширений и настроек пользователя от каталогов `~/.vscode` или `~/.void`.
*   **Замена графических ассетов:** Замена иконок сборки (`.ico`, `.icns`), сплэш-скринов и логотипов во внутреннем коде Workbench на фирменную символику проекта Nexus.

### Этап 2: Блокировка прямой генерации и Перехват Запросов
**Цель:** Интеграция ТРИЗ-фильтра в кодовую базу редактора.

*   Модификация сервиса обработки ИИ-запросов (`src/vs/workbench/contrib/void-ai/`).
*   Отключение дефолтных внешних AI-прокси Void.
*   Реализация перехватчика (Middleware): при отправке промпта через интерфейс чата или инлайн-инструкции, текст принудительно направляется на локальный порт MCP-сервера Nexus.

### Этап 3: Разработка логического ядра TRIZ MCP Server
**Цель:** Создание инструментария для пошагового прохождения АРИЗ-85-В.

*   Разработка функций FastMCP:
    *   `analyze_architectural_conflict(code_context, task)`: возвращает техническое и физическое противоречие задачи.
    *   `evaluate_ikr(conflict)`: генерирует формулировку Идельного Конечного Результата.
    *   `suggest_triz_principles(conflict_matrix)`: сопоставляет проблему с массивом 40 приемов и выдает 2 оптимальных метода рефакторинга/написания кода.
*   Подключение базы знаний через `MemoryBridge` для обеспечения LLM точными выдержками из трудов Альтшуллера.

### Этап 4: Разработка Инженерного Веб-интерфейса (TRIZ Log Panel)
**Цель:** Создание UI-компонента для управления изобретательским циклом.

*   Создание кастомной боковой панели (Webview).
*   Внедрение трех обязательных полей валидации перед применением кода:
    1.  **Поле противоречия:** С какими ограничениями столкнулась система (например, Скорость vs Память).
    2.  **Поле ИКР:** Описание идеального состояния кода (функция выполняется, сущности нет).
    3.  **Примененный паттерн ТРИЗ:** Инженерное обоснование выбранного подхода (Динамизация, Дробление, Инверсия).
*   Кнопка **«Принять инженерный паспорт (Approve & Code)»** — только после нажатия данные передаются конечному исполнителю (Claude Code) для генерации diff-файла.

2. **Суверенность данных:** Ни одна часть анализируемого исходного кода не должна отправляться на сторонние сервера для индексации или обучения. Все ТРИЗ-картотеки и логи АРИЗ обрабатываются локальными инференс-моделями или изолированными API.
